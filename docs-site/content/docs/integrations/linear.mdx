Linear is the primary issue tracking integration. Triage creates, updates, and searches Linear issues using the official `@linear/sdk` client. A webhook from Linear to the runtime triggers the resolution review phase of the triage workflow.

## Configuration

```bash
LINEAR_API_KEY=lin_api_...
```

The key is optional at startup — the runtime instantiates a `LinearClient` singleton only if the key is set:

```typescript
const linearClient = config.LINEAR_API_KEY
  ? new LinearClient({ apiKey: config.LINEAR_API_KEY })
  : null;
```

All endpoints that require Linear return `{ success: false, error: { code: 'NO_LINEAR_KEY' } }` if the key is not set.

## Team Configuration

Issue creation and queries are scoped to a single team defined in `LINEAR_CONSTANTS`:

```typescript
export const LINEAR_CONSTANTS = {
  TEAM_ID: '645a639b-39e2-4abe-8ded-3346d2f79f9f',   // triage-hackathon team
  // ...
}
```

## The Five Tools

### `createLinearIssueTool`

Creates a new Linear issue. This is the only tool that requires user approval before being called by the orchestrator — the user must explicitly approve the triage card first.

**Creates:** A new issue in the `Triage` state with appropriate severity label, title, description (full triage output), and assignee.

**Fallback:** If `LINEAR_API_KEY` is not set or the API call fails, the issue is written to the `local_tickets` LibSQL table instead.

### `updateLinearIssueTool`

Updates an existing Linear issue. Used to change state, assignee, labels, or priority.

### `getLinearIssueTool`

Fetches a specific Linear issue by ID. Used by the resolution reviewer to retrieve the original triage data when verifying a fix.

### `listLinearIssuesTool` / `searchLinearIssues`

Searches and lists issues filtered by team, state, or text. Used by the orchestrator for duplicate detection before running triage.

### `getTeamMembersTool`

Returns active, non-guest team members. Used by the orchestrator to suggest assignees for new incidents.

## API Routes

Three Linear-related routes are registered directly on the Mastra server:

### `GET /api/linear/issues`

Fetches all team issues and groups them by state name. Returns:

```json
{
  "success": true,
  "data": {
    "Triage": [...],
    "In Progress": [...],
    "Done": [...]
  }
}
```

Each issue object includes `id`, `identifier`, `title`, `priority`, `estimate`, `project`, `url`, `createdAt`, `updatedAt`, `assignee`, and `labels`.

### `GET /api/linear/cycle/active`

Returns the currently active cycle with progress metrics:

```json
{
  "success": true,
  "data": {
    "id": "...",
    "name": "Cycle 5",
    "number": 5,
    "startsAt": "2026-04-07T00:00:00.000Z",
    "endsAt": "2026-04-14T00:00:00.000Z",
    "progress": 0.43,
    "scopeCount": 12,
    "completedScopeCount": 5,
    "startedScopeCount": 3
  }
}
```

### `GET /api/linear/members`

Returns active, non-guest team members:

```json
{
  "success": true,
  "data": {
    "members": [
      { "id": "...", "name": "Fernando", "email": "...", "displayName": "Fernando" }
    ]
  }
}
```

## Webhook Flow

### Setting Up the Webhook

Register the Linear webhook via the setup endpoint:

```bash
curl -X POST http://localhost:4111/api/linear/webhook/setup \
  -H "Content-Type: application/json" \
  -d '{"url": "https://your-public-url.example/api/webhooks/linear"}'
```

This registers a webhook on your Linear team for `Issue` resource type events.

### Receiving Webhooks

`POST /api/webhooks/linear` receives Linear webhook payloads. The handler:

1. Filters for `action: "update"`, `type: "Issue"` events only
2. Checks if the new state has `type: "completed"` — skips all others
3. Searches suspended workflow runs in LibSQL for one matching the `issueId`
4. If found, resumes the workflow at the `suspend` step with `{ newStatus, updatedAt }`
5. Responds immediately (resume runs in the background)

The workflow suspension mechanism stores the `issueId` in the ticket step output, embedded in the workflow snapshot in LibSQL. The webhook handler parses this snapshot to find the matching run.
