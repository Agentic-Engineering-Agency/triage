## Implementation Issues

### Critical (must fix)

1. **REDIS HEALTHCHECK WILL FAIL WITH AUTH ENABLED.** The redis service uses `--requirepass ${REDIS_AUTH}` but the healthcheck is `redis-cli ping`. When requirepass is set, unauthenticated commands return `NOAUTH Authentication required`. The healthcheck MUST be:
   ```yaml
   test: ["CMD", "redis-cli", "-a", "${REDIS_AUTH}", "ping"]
   ```
   Or use the `REDISCLI_AUTH` environment variable. **Without this fix, redis will never report healthy, and langfuse-worker will never start.** This is a DEMO BLOCKER.

2. **MinIO HEALTHCHECK MAY FAIL.** The compose uses `cgr.dev/chainguard/minio` (Chainguard image) with healthcheck `mc ready local`. The Chainguard MinIO image is a minimal distroless-style image and may NOT include the `mc` (MinIO Client) binary. Standard `minio/minio` images include `mc`. Verify this works or switch to `minio/minio:latest` or use `curl -f http://localhost:9000/minio/health/ready` instead.

3. **LANGFUSE_S3_MEDIA_UPLOAD_ENDPOINT is wrong for internal Docker use.** Line 28 of docker-compose.yml sets `LANGFUSE_S3_MEDIA_UPLOAD_ENDPOINT: http://localhost:9090`. Inside the Docker network, `localhost` resolves to the container itself, NOT the host. This should be `http://minio:9000` (same as the EVENT_UPLOAD_ENDPOINT). Media uploads will fail at runtime.

4. **Missing LANGFUSE_S3_BATCH_EXPORT variables in docker-compose.yml.** The .env.example defines `LANGFUSE_S3_BATCH_EXPORT_SECRET_ACCESS_KEY` but the langfuse-worker-env anchor has NO batch export S3 configuration (BUCKET, PREFIX, REGION, ENDPOINT, ACCESS_KEY_ID, SECRET_ACCESS_KEY, FORCE_PATH_STYLE). Langfuse batch export will be non-functional.

### Warnings (should fix)

5. **ClickHouse has no authentication configured.** The langfuse services send `CLICKHOUSE_PASSWORD=${CLICKHOUSE_PASSWORD}` and `CLICKHOUSE_USER=default` but the clickhouse service itself has NO environment variables setting up authentication. The default ClickHouse `default` user has no password. This means either: (a) the password env vars are ignored by Langfuse (unlikely), or (b) Langfuse will fail to connect if a password is expected. **Verify the Langfuse docker-compose reference to determine if ClickHouse auth is actually needed.**

6. **Floating image tags create reproducibility risk:**
   - `clickhouse/clickhouse-server` — no tag, resolves to `:latest`
   - `ghcr.io/tursodatabase/libsql-server:latest` — explicitly `:latest`
   - `cgr.dev/chainguard/minio` — no tag
   Pin these to specific versions before the hackathon demo.

7. **libsql port 5001 exposed to 0.0.0.0.** The compose has `5001:5001` with no 127.0.0.1 prefix. This exposes the gRPC port publicly. The spec (REQ-D05) says only ports 3001, 3000, 9090, 8080, and optionally 4111 should be publicly bound. Fix: `127.0.0.1:5001:5001` or remove.

8. **runtime port 4111 exposed to 0.0.0.0.** The spec says this MAY be exposed for debugging. Current: `4111:4111`. Consider binding to `127.0.0.1:4111:4111` since Caddy proxies all API traffic. This is acceptable per spec but weakens the single-origin architecture claim.

9. **Dockerfile HEALTHCHECK duplication.** Both Dockerfile.runtime (line 15) and Dockerfile.frontend (line 14) define HEALTHCHECK directives. The docker-compose.yml ALSO defines healthcheck for both services. The compose version overrides the Dockerfile version, making the Dockerfile HEALTHCHECKs dead code. Remove from Dockerfiles to avoid confusion.

10. **clickhouse service has no environment section at all.** If Langfuse expects CLICKHOUSE_PASSWORD to authenticate, this is a runtime failure. If not, document why.

## Security Findings

### Critical

1. **Redis port 6379 accessible without auth via healthcheck bypass.** While the port is bound to 127.0.0.1 (good), the healthcheck uses unauthenticated `redis-cli ping`. More importantly, if the healthcheck is fixed to include the password, the password will be visible in `docker inspect` output and `docker compose config` output. Consider using `REDISCLI_AUTH` env var approach instead.

2. **libsql gRPC port 5001 publicly exposed (0.0.0.0).** No authentication on libsql means anyone on the network can connect to the database via gRPC. This is a data exposure risk.

### Warnings

3. **.env.example is safe to commit.** All 17 secret values use CHANGEME placeholders. No real secrets detected. PASS.

4. **No Content-Security-Policy header in Caddyfile.** The spec requires HSTS, X-Content-Type-Options, X-Frame-Options, and Referrer-Policy (all present). CSP is not in scope but would strengthen security. Not required by spec.

5. **ClickHouse has no authentication.** The default user has unrestricted access. Combined with 127.0.0.1 binding, the blast radius is limited to the host machine, but any process on the host can read/write ClickHouse data.

6. **No resource limits (cpu/memory) on Docker containers.** A runaway container (e.g., ClickHouse under load) could consume all host resources and crash other services. Consider adding `deploy.resources.limits` in compose.

7. **Dockerfile.runtime COPY . . copies everything including .env.** The builder stage does `COPY . .` which includes .env if present. While the production stage only copies `.mastra/output/`, secrets could leak into Docker layer cache. Add `.env` to `.dockerignore`.

## Spec Ambiguity Issues

1. **REQ-D05 contradicts itself on minio port.** Spec says "minio:9090 (S3 API)" but MinIO S3 API runs on port 9000 internally. Port 9090 is the HOST mapping to container port 9000. The spec should clarify this is the host-side mapping. Tests correctly check for "9090" in ports string.

2. **REQ-D11 acceptance criteria are untestable with static analysis.** "GIVEN a fresh MinIO container... WHEN it reaches healthy status THEN the langfuse bucket SHALL exist" requires a running Docker environment. No static test can verify this. The docker-compose.yml handles it via `mkdir -p /data/langfuse` in the minio command, but this is NOT the same as creating an S3 bucket — it creates a filesystem directory. MinIO auto-detects directories as buckets, so it works, but it's worth noting.

3. **REQ-D13 is fundamentally untestable without running services.** Graceful degradation requires integration testing with actual containers. The current static tests are justified placeholders but should be explicitly marked as "requires CI/manual" rather than passing silently.
