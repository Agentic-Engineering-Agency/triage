## 1. Purpose

This spec defines how the Triage project integrates with Langfuse for observability of LLM-powered triage workflows. The primary goal is end-to-end tracing of OpenRouter AI calls with minimal code changes. This spec covers the chosen integration path, environment and network requirements, trace verification, Cloudflare tunnel setup for exposing Langfuse to OpenRouter, and Docker Compose updates.

---

## 2. Scope

### 2.1 In Scope

- OpenRouter Broadcast integration (preferred, zero code change)
- Langfuse SDK integration as fallback for direct Mastra instrumentation
- Environment variable verification (LANGFUSE_PUBLIC_KEY, LANGFUSE_SECRET_KEY, LANGFUSE_BASEURL)
- Network path verification (runtime → langfuse-web:3000)
- Cloudflare Tunnel setup for exposing Langfuse to OpenRouter
- Trace verification test (end-to-end triage call)
- AGENTS_USE.md observability section update

### 2.2 Out of Scope

- Kubernetes Helm chart updates (langfuse infrastructure already has separate Helm templates; k8s path documented in SCALING.md)
- Langfuse server-side configuration (org/project creation, user management)
- Non-OpenRouter LLM tracing (e.g., direct Anthropic/OpenAI calls — not used in this stack)

---

## 3. Integration Path Decision

### Chosen Path: OpenRouter Broadcast

**Rationale:**

1. Zero code changes required in the runtime
2. All OpenRouter API calls are automatically traced with no SDK modifications
3. No `langfuse` npm package dependency in the runtime
4. Works with the existing `createOpenRouter()` setup in orchestrator.ts
5. OpenRouter provides a dashboard UI for configuration

**Fallback:** Langfuse SDK direct instrumentation in the runtime (using `@langfuse/sdk` with AI SDK v1.3.3 instrumentation hooks) — only if OpenRouter Broadcast proves unreliable or if custom span naming is required.

## 4. Requirements

### REQ-OBS-01 [P0] — OpenRouter Broadcast Setup

The system SHALL use OpenRouter Broadcast to automatically trace all LLM calls made through OpenRouter.

**Acceptance Criteria:**
- Langfuse API keys (public + secret) are present in `.env`
- LANGFUSE_BASEURL is set to the Cloudflare tunnel URL for self-hosted Langfuse (`https://langfuse.agenticengineering.lat`)
- OpenRouter dashboard Settings → Observability → Broadcast is enabled with the correct keys
- A test API call through OpenRouter produces a visible trace in Langfuse within 60 seconds

**Scenarios:**
1. **Happy path:** User sends an incident report; trace appears in Langfuse with model, latency, token usage, and trace_id
2. **No keys:** If LANGFUSE_SECRET_KEY is absent, OpenRouter ignores the broadcast silently (no error); no trace created
3. **Wrong base URL:** If LANGFUSE_BASEURL points to an unreachable host, OpenRouter fails the test connection gracefully

The runtime container SHALL be able to reach langfuse-web at `http://langfuse-web:3000` over the `langfuse` Docker network.

- `docker exec <runtime-container> curl -sf http://langfuse-web:3000/api/public/health` returns HTTP 200

**Scenarios:**
1. **Internal access:** Langfuse web UI accessible inside Docker network at `langfuse-web:3000`
2. **External access:** Langfuse web UI accessible outside Docker via Cloudflare Tunnel at `https://langfuse.agenticengineering.lat`

All required variables exist in `.env` and are validated in `runtime/src/lib/config.ts`:

| Variable | Current Status | Required Action |
|---|---|---|
| `LANGFUSE_PUBLIC_KEY` | Set as `LANGFUSE_INIT_PROJECT_PUBLIC_KEY` | Generate real key pair (see Section 5.1) |
| `LANGFUSE_SECRET_KEY` | Set | Generate real key pair (see Section 5.1) |
| `LANGFUSE_BASEURL` | `http://langfuse-web:3000` | Update to `https://langfuse.agenticengineering.lat` after tunnel is running |
| `ENCRYPTION_KEY` | Fixed (64 hex chars) | No action needed |
| `SALT` | Fixed (64 hex chars) | No action needed |
| `NEXTAUTH_SECRET` | Fixed (64 hex chars) | No action needed |

### 5.1 Generating Langfuse API Keys

Langfuse uses a specific key format. Generate keys using:

```bash
# Public key format: pk_<32-char-hex>
# Secret key format: sk_<32-char-hex>
python3 -c "import secrets; print(f'pk_{secrets.token_hex(16)}'); print(f'sk_{secrets.token_hex(16)}')"
```

Example output:
```
pk_e91cd15cb0ddcd78cf3083d67ded03e8
sk_62e53f0a1b2c3d4e5f67890123456789
```

Then set in `.env`:
```
LANGFUSE_INIT_PROJECT_PUBLIC_KEY=pk_<generated>
LANGFUSE_INIT_PROJECT_SECRET_KEY=sk_<generated>
```

### 5.2 Langfuse Admin Password

The initial admin password must be set to a strong value. To change it, update `LANGFUSE_INIT_USER_PASSWORD` in `.env` and restart the Langfuse containers (the init user is only created on first start if no users exist in the database).

**Current DB state:** Admin user exists with hashed password. To reset:
1. Drop the `users` table or delete the admin user row
2. Restart langfuse-web — it will re-create from `LANGFUSE_INIT_USER_*` env vars
3. Or use the Langfuse UI "forgot password" flow (requires email configuration)

### OpenRouter Broadcast Configuration

**Cloudflare Tunnel URL:** `https://langfuse.agenticengineering.lat`

1. Log in to OpenRouter at `https://openrouter.ai`
2. Navigate to **Settings → Observability → Enable Broadcast → Langfuse**
3. Enter:
   - **Secret Key:** `LANGFUSE_SECRET_KEY` value from `.env`
   - **Public Key:** `LANGFUSE_PUBLIC_KEY` value from `.env`
   - **Base URL:** `https://langfuse.agenticengineering.lat` (Cloudflare Tunnel URL)
4. Click **Test Connection** — saves only on success
5. Send a test API request via OpenRouter; trace appears in Langfuse within 60 seconds

### Custom Trace Metadata (Optional)

To attach custom `trace_id` and `span_name` to OpenRouter requests, use the `trace` field in API requests:

```json
{
  "model": "inception/mercury-2",
  "messages": [{"role": "user", "content": "Test incident"}],
  "trace": {
    "trace_id": "workflow_12345",
    "trace_name": "Triage Pipeline",
    "span_name": "Orchestrator Agent",
    "generation_name": "Mercury Inference",
    "environment": "development"
  }
}
```

This is optional — OpenRouter generates a trace_id automatically if not provided.

## 8. Langfuse SDK Fallback (Fallback Path)

If OpenRouter Broadcast is insufficient, add direct Langfuse SDK instrumentation to the runtime:

```bash
cd runtime
npm install @langfuse/sdk langfuse-core
```

Initialize Langfuse client in `runtime/src/lib/langfuse.ts`:

```typescript
import { Langfuse } from '@langfuse/sdk';

let langfuse: Langfuse | undefined;

export function getLangfuseClient() {
  if (!langfuse) {
    const publicKey = process.env.LANGFUSE_PUBLIC_KEY;
    const secretKey = process.env.LANGFUSE_SECRET_KEY;
    const baseUrl = process.env.LANGFUSE_BASEURL || 'http://langfuse-web:3000';

    if (!publicKey || !secretKey) {
      console.warn('[Langfuse] Keys not set — tracing disabled');
      return undefined;
    }

    langfuse = new Langfuse({
      publicKey,
      secretKey,
      baseUrl,
    });
  }
  return langfuse;
}
```

In instrument AI SDK in `runtime/src/mastra/index.ts`:

```typescript
import { withLangfuseTracing } from '@langfuse/sdk/ai-sdk';

const langfuse = getLangfuseClient();
if (langfuse) {
  // Wrap model or agent with Langfuse tracing
  // See: https://langfuse.com/docs/integrations/langchain/python
}
```
