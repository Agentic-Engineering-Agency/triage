# SPEC-20260409-001: Db Schema Drizzle Integration

**ID:** SPEC-20260409-001

**Updated:** 2026-04-09

## Purpose

Consolidate and implement the complete database layer for the Triage SRE agent. This spec covers Drizzle ORM integration with LibSQL/Turso, Better Auth session management, wiki RAG vector storage with F32_BLOB DiskANN search, local fallback ticket storage, and all supporting infrastructure (scripts, env vars, Docker verification). The database serves four roles: Mastra workflow state, authentication, wiki vector search, and graceful-degradation ticket storage.

## Scope

### In Scope

- Drizzle ORM client configuration (`drizzle-orm/libsql` + `@libsql/client`)
- `drizzle.config.ts` with dialect `turso` and schema path `runtime/src/db/schema.ts`
- 4 auth tables (`auth_user`, `auth_session`, `auth_account`, `auth_verification`) with `auth_` prefix
- Better Auth instance with Drizzle adapter (provider: `sqlite`)
- Mastra route mounting: `registerApiRoute('/auth/*')` for Better Auth
- Zod schemas derived from Drizzle tables via `drizzle-orm/zod`
- `wiki_documents` table for ingested codebase documents
- `wiki_chunks` table with `F32_BLOB(1536)` vector column for DiskANN similarity search
- `local_tickets` table for graceful-degradation when Linear is unavailable
- Drizzle relations between wiki tables and between tickets and auth users
- `db:push` and `db:studio` npm scripts in `runtime/package.json`
- Docker Compose libsql service verification
- `.env.example` consolidation with all integration env vars
- CLAUDE.md updates for new drizzle-kit commands
- Unit tests for client, schema, and auth
- Integration tests for push, vector ops, and CRUD

### Out of Scope

- Drizzle migration files (`generate` + `migrate`) — deferred to production readiness spec
- Better Auth UI components — frontend spec concern
- Wiki document ingestion pipeline — workflow spec concern
- Linear webhook handler — separate spec (SPEC-20260408-002 scope)
- Mastra LibSQLStore configuration changes — already working, not modified
- Turso cloud deployment — local sqld Docker only for hackathon
- Auth session middleware / route guards — frontend + auth spec concern

## Dependencies

### External Packages (to be added to `runtime/package.json`)

| Package | Purpose |
|---------|---------|
| `drizzle-orm` | ORM core + LibSQL dialect + Zod integration |
| `@libsql/client` | LibSQL/Turso client driver |
| `drizzle-kit` | Schema push, studio, introspect (devDependency) |
| `better-auth` | Authentication framework |
| `@better-auth/drizzle-adapter` | Better Auth ↔ Drizzle ORM bridge |

### Internal Dependencies

- Docker Compose libsql service (sqld v0.24.32) — MUST be running and healthy
- `.env` with `LIBSQL_URL`, `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`
- Mastra instance at `runtime/src/mastra/index.ts` — for route registration
- Existing Zod schemas at `runtime/src/lib/schemas/wiki.ts` — alignment target

### Turso/LibSQL Gotchas (Normative)

1. `drizzle.config.ts` dialect MUST be `'turso'` — NOT `'sqlite'` — even for local sqld
2. Runtime import MUST use `drizzle-orm/libsql` — NOT `drizzle-orm/turso`
3. Better Auth adapter provider MUST be `'sqlite'` — NOT `'turso'`
4. Zod integration MUST use `drizzle-orm/zod` — NOT the deprecated `drizzle-zod` package
5. sqld Docker image MUST set `platform: linux/amd64`
6. All auth tables MUST use `auth_` prefix to avoid Mastra's built-in `account` table conflict
7. `drizzle-kit push` MUST be used for development; `generate` + `migrate` deferred to prod

### Phase 1: Foundation

#### REQ-DB01: Drizzle Client Module (P0)

The system SHALL provide a Drizzle ORM client module at `runtime/src/db/client.ts`.

The client MUST:
- Import `drizzle` from `drizzle-orm/libsql`
- Import `createClient` from `@libsql/client`
- Create a libsql client using `process.env.LIBSQL_URL` with fallback `'http://libsql:8080'`
- Export a named `db` constant created via `drizzle(client, { logger })` where logger is enabled when `NODE_ENV === 'development'`
- Export the raw `client` for cases requiring direct libsql access

**Acceptance Criteria:**

**Scenario 1 — Happy Path: Client creation with default URL**
```
GIVEN no LIBSQL_URL environment variable is set
WHEN the db module is imported
THEN a Drizzle client SHALL be created with URL 'http://libsql:8080'
AND the exported db object SHALL be a valid Drizzle instance
```

**Scenario 2 — Edge Case: Custom URL from environment**
```
GIVEN LIBSQL_URL is set to 'http://localhost:9090'
WHEN the db module is imported
THEN the underlying libsql client SHALL use 'http://localhost:9090'
```

**Scenario 3 — Error Case: Malformed URL**
```
GIVEN LIBSQL_URL is set to an invalid string (e.g., 'not-a-url')
WHEN the db module is imported and a query is attempted
THEN the client SHALL throw a connection error
AND the error SHALL be catchable by callers
```

#### REQ-DB03: Auth Schema Tables (P0)

The system SHALL define four authentication tables in `runtime/src/db/schema.ts` using `sqliteTable` from `drizzle-orm/sqlite-core`.

All tables MUST use the `auth_` prefix. The tables are:

**auth_user:**
- `id` — text, primary key
- `name` — text, not null
- `email` — text, not null, unique
- `emailVerified` — integer (boolean), not null, default false
- `image` — text (nullable)
- `createdAt` — integer (unix timestamp), not null
- `updatedAt` — integer (unix timestamp), not null

**auth_session:**
- `id` — text, primary key
- `expiresAt` — integer, not null
- `token` — text, not null, unique
- `ipAddress` — text (nullable)
- `userAgent` — text (nullable)
- `userId` — text, not null, foreign key → auth_user.id

**auth_account:**
- `id` — text, primary key
- `accountId` — text, not null
- `providerId` — text, not null
- `userId` — text, not null, foreign key → auth_user.id
- `accessToken` — text (nullable)
- `refreshToken` — text (nullable)
- `idToken` — text (nullable)
- `accessTokenExpiresAt` — integer (nullable)
- `refreshTokenExpiresAt` — integer (nullable)
- `scope` — text (nullable)
- `password` — text (nullable)
- `createdAt` — integer, not null
- `updatedAt` — integer, not null

**auth_verification:**
- `id` — text, primary key
- `identifier` — text, not null
- `value` — text, not null
- `expiresAt` — integer, not null
- `createdAt` — integer (nullable)
- `updatedAt` — integer (nullable)

**Acceptance Criteria:**

**Scenario 1 — Happy Path: All four tables defined**
```
GIVEN runtime/src/db/schema.ts is imported
WHEN the exported tables are inspected
THEN auth_user, auth_session, auth_account, and auth_verification SHALL each be valid sqliteTable definitions
AND each table name SHALL start with 'auth_'
```

**Scenario 2 — Edge Case: Foreign key integrity**
```
GIVEN auth_session table definition
WHEN the userId column is inspected
THEN it SHALL reference auth_user.id as a foreign key
AND auth_account.userId SHALL also reference auth_user.id
```

**Scenario 3 — Error Case: Table name collision avoidance**
```
GIVEN the Mastra framework uses a built-in 'account' table
WHEN schema.ts is loaded alongside Mastra
THEN no table named 'account' (without prefix) SHALL exist in the schema
AND auth_account SHALL be the only account-related table
```

#### REQ-DB08: Wiki Chunks Table with Vector Column (P0)

The system SHALL define a `wiki_chunks` table in `runtime/src/db/schema.ts` with native vector support.

The table MUST contain:
- `id` — text, primary key
- `documentId` — text, not null, foreign key → wiki_documents.id
- `content` — text, not null
- `chunkIndex` — integer, not null
- `embedding` — F32_BLOB(1536) using a custom Drizzle type, nullable (null until embedding is computed)
- `createdAt` — integer (unix timestamp), not null, default current timestamp

The custom type `float32Array` MUST:
- Be defined using `customType` from `drizzle-orm/sqlite-core`
- Accept a `dimensions` config parameter
- Map `dataType` to `F32_BLOB(${dimensions})`
- Implement `fromDriver` to convert Buffer → number[] via Float32Array
- Implement `toDriver` to convert number[] → `sql\`vector32(${JSON.stringify(value)})\``

**Acceptance Criteria:**

**Scenario 1 — Happy Path: Vector column definition**
```
GIVEN wiki_chunks table is defined with the float32Array custom type
WHEN the embedding column's SQL type is generated
THEN it SHALL produce 'F32_BLOB(1536)'
AND the column SHALL be nullable
```

**Scenario 2 — Edge Case: Vector serialization round-trip**
```
GIVEN a 1536-dimensional float array [0.1, 0.2, ..., 0.0]
WHEN toDriver is called
THEN it SHALL produce a sql`vector32(...)` expression
AND when fromDriver receives the stored Buffer
THEN the reconstructed array SHALL match the original within floating-point precision
```

**Scenario 3 — Error Case: Wrong dimension count**
```
GIVEN the embedding column is configured for 1536 dimensions
WHEN an array of 768 dimensions is passed to toDriver
THEN the SQL expression SHALL still be generated (dimension check is at DB level)
AND sqld SHALL reject the insert with a vector dimension mismatch error
```

#### REQ-DB09: Local Tickets Table (P0)

The system SHALL define a `local_tickets` table in `runtime/src/db/schema.ts` for graceful degradation when Linear is unavailable.

The table MUST contain:
- `id` — text, primary key
- `linearIssueId` — text, nullable (populated after successful Linear sync)
- `title` — text, not null
- `description` — text, not null
- `severity` — text, not null (values: 'critical', 'high', 'medium', 'low')
- `priority` — integer, not null (1–4, where 1 is urgent)
- `status` — text, not null, default 'triage' (values align with Linear workflow states)
- `assigneeId` — text, nullable, foreign key → auth_user.id
- `createdAt` — integer (unix timestamp), not null, default current timestamp
- `updatedAt` — integer (unix timestamp), not null, default current timestamp
- `syncedAt` — integer (unix timestamp), nullable (null until synced to Linear)

**Acceptance Criteria:**

**Scenario 1 — Happy Path: Ticket creation without Linear**
```
GIVEN Linear is unavailable
WHEN a ticket is inserted into local_tickets with linearIssueId = null
THEN the row SHALL be created successfully
AND syncedAt SHALL be null
AND status SHALL default to 'triage'
```

**Scenario 2 — Edge Case: Ticket synced to Linear**
```
GIVEN a local_tickets row with linearIssueId = null
WHEN Linear becomes available and the ticket is synced
THEN linearIssueId SHALL be updated to the Linear issue identifier
AND syncedAt SHALL be set to the sync timestamp
```

**Scenario 3 — Error Case: Invalid severity value**
```
GIVEN the severity column is text type (no DB-level enum constraint)
WHEN an invalid severity like 'extreme' is inserted
THEN the DB SHALL accept the row (no DB constraint)
AND application-level Zod validation MUST reject invalid severity values before DB insert
```

#### REQ-DB10: Drizzle Relations (P1)

The system SHALL define Drizzle relations in `runtime/src/db/schema.ts` for all foreign key relationships.

Relations MUST include:
- `wiki_chunks` → `wiki_documents` (many-to-one via `documentId`)
- `wiki_documents` → `wiki_chunks` (one-to-many)
- `auth_session` → `auth_user` (many-to-one via `userId`)
- `auth_account` → `auth_user` (many-to-one via `userId`)
- `local_tickets` → `auth_user` (many-to-one via `assigneeId`, optional)

Relations MUST be defined using `relations()` from `drizzle-orm` and exported from the schema module.

**Acceptance Criteria:**

**Scenario 1 — Happy Path: Relation query**
```
GIVEN relations are defined between wiki_chunks and wiki_documents
WHEN a relational query fetches a chunk with its parent document
THEN the document data SHALL be included in the result
AND the query SHALL use a join (not N+1)
```

**Scenario 2 — Edge Case: Optional relation**
```
GIVEN local_tickets.assigneeId is nullable
WHEN a ticket with assigneeId = null is queried with relations
THEN the assignee relation SHALL resolve to null
AND the query SHALL NOT fail
```

**Scenario 3 — Error Case: Orphaned foreign key**
```
GIVEN a wiki_chunks row references a documentId that does not exist in wiki_documents
WHEN a relational query is attempted
THEN the query SHOULD return null for the document relation
AND the application layer SHOULD handle missing parent documents gracefully
```

## Technical Approach

### Architecture

```
runtime/src/
├── db/
│   ├── client.ts          # Drizzle + libsql client (REQ-DB01)
│   ├── client.test.ts     # Unit tests (REQ-DB14)
│   ├── schema.ts          # All table definitions (REQ-DB03, DB07, DB08, DB09, DB10)
│   ├── schema.test.ts     # Unit tests (REQ-DB15)
│   ├── zod-schemas.ts     # createInsertSchema/createSelectSchema (REQ-DB06)
│   └── custom-types.ts    # float32Array custom type (used by REQ-DB08)
├── lib/
│   ├── auth.ts            # Better Auth instance (REQ-DB04)
│   ├── auth.test.ts       # Unit tests (REQ-DB16)
│   └── schemas/
│       └── wiki.ts        # Existing API-level Zod schemas (not modified)
└── mastra/
    └── index.ts           # Updated with auth route (REQ-DB05)

drizzle.config.ts              # Drizzle Kit config (REQ-DB02)
.env.example                   # Consolidated env vars (REQ-DB12, DB21)
CLAUDE.md                      # Updated docs (REQ-DB20)

tests/db-integration/
├── schema-push.test.ts    # Integration: push verification (REQ-DB17)
├── wiki-vectors.test.ts   # Integration: vector ops (REQ-DB18)
└── local-tickets.test.ts  # Integration: CRUD (REQ-DB19)
```

### Key Design Decisions

1. **Single schema file:** All tables in one `schema.ts` file — Drizzle Kit requires a single schema entry point, and the total table count (7) is manageable.

2. **Custom type isolation:** The `float32Array` custom type is in `custom-types.ts` to keep schema.ts readable and allow reuse if other vector columns are added.

3. **Zod schema separation:** DB-derived Zod schemas (`drizzle-orm/zod`) coexist with hand-written API schemas (`runtime/src/lib/schemas/`). They serve different purposes — DB schemas for ORM validation, API schemas for HTTP I/O. No forced unification.

4. **Auth prefix strategy:** All Better Auth tables use `auth_` prefix (not configurable at runtime) to avoid Mastra's internal `account` table collision. This is hardcoded in schema definitions.

5. **Integration test isolation:** DB integration tests live in `tests/db-integration/` (top-level), not co-located, because they require Docker infrastructure and have different lifecycle from unit tests.

### Technology Stack

| Component | Technology | Version/Notes |
|-----------|-----------|---------------|
| ORM | drizzle-orm | Latest, with /libsql dialect |
| DB Client | @libsql/client | HTTP protocol to sqld |
| DB Server | sqld (libsql-server) | v0.24.32, Docker linux/amd64 |
| Auth | better-auth | With drizzle-adapter, provider sqlite |
| Schema Tooling | drizzle-kit | push (dev), generate+migrate (future) |
| Vector Search | Native F32_BLOB | DiskANN via sqld, 1536 dims (OpenAI ada-002) |
| Zod Bridge | drizzle-orm/zod | createInsertSchema, createSelectSchema |
