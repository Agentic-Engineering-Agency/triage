## Purpose

Implement session-based authentication for the Triage SRE agent using Better Auth with a Drizzle ORM adapter against LibSQL self-hosted (sqld v0.24.32 in Docker). Covers: Drizzle client singleton, auth schema tables, drizzle-kit push workflow, Better Auth instance with drizzleAdapter, Zod validators, and route mounting in Mastra. Frontend auth client is OUT OF SCOPE (separate ticket FE-03).

---

## Scope

### In Scope
- Drizzle client singleton (`runtime/src/db/client.ts`) using `drizzle-orm/libsql` + `@libsql/client`
- `drizzle.config.ts` with `dialect: 'turso'` at project root
- Auth schema tables: `auth_user`, `auth_session`, `auth_account`, `auth_verification` in `runtime/src/db/schema.ts` (prefixed to avoid conflict with Mastra's existing `account` table)
- `drizzle-kit push` to apply schema (no migration files)
- Better Auth instance with `drizzleAdapter(db, { provider: 'sqlite' })` in `runtime/src/auth/index.ts`
- Session config: 7-day fixed expiry, no sliding window, HttpOnly + SameSite=Lax + Secure in prod
- `trustedOrigins` including `http://localhost:3001` (dev) and production origin
- `/auth/*` route mounting in Mastra (sign-in, sign-up, sign-out, session)
- Zod schemas for auth tables via `drizzle-orm/zod` (NOT `drizzle-zod`)
- `BETTER_AUTH_SECRET` (min 32 chars) and `BETTER_AUTH_URL` in `.env.example`
- Packages added to existing `runtime/package.json`: `better-auth`, `@better-auth/drizzle-adapter`, `drizzle-kit`, `@libsql/client`, `drizzle-orm`

### Out of Scope
- Frontend auth client (`frontend/src/lib/auth-client.ts`) — FE-03
- Frontend `useAuth` hook replacement — FE-03
- Frontend route guards — FE-03
- OAuth/social providers (email+password only)
- Two-factor authentication
- Password reset email flows
- Non-Drizzle database adapters

## Requirements

### REQ-A01: Drizzle Client Singleton

**Priority:** P0
**Description:** The runtime SHALL provide a singleton Drizzle client connected to LibSQL at `http://libsql:8080` using `drizzle-orm/libsql`.

#### Acceptance Criteria
- **GIVEN** the runtime container starts **WHEN** `runtime/src/db/client.ts` is imported **THEN** a Drizzle client connected to LibSQL via `@libsql/client` HTTP protocol SHALL be exported
- **GIVEN** `LIBSQL_URL` env var is set to `http://libsql:8080` **WHEN** the client initializes **THEN** it SHALL connect to that URL without an authToken

#### Scenarios
- **Happy path:** Client initializes successfully, queries execute against LibSQL
- **Edge case:** `LIBSQL_URL` env var is missing → falls back to default `http://libsql:8080`
- **Error case:** LibSQL server unreachable → connection throws, application logs error

---

### REQ-A02: drizzle.config.ts with turso Dialect

**Priority:** P0
**Description:** The project SHALL contain a `drizzle.config.ts` using `dialect: 'turso'` pointing to the LibSQL URL.

#### Acceptance Criteria
- **GIVEN** `drizzle.config.ts` exists at the project root **WHEN** `npx drizzle-kit push` runs **THEN** it SHALL use `dialect: 'turso'`
- **GIVEN** the schema file is at `runtime/src/db/schema.ts` **WHEN** drizzle-kit parses the config **THEN** it SHALL find all exported tables

#### Scenarios
- **Happy path:** `drizzle-kit push` connects to LibSQL and applies schema without errors
- **Edge case:** `LIBSQL_URL` env var points to wrong URL → push fails with connection error
- **Error case:** Schema has syntax errors → drizzle-kit reports parse error and exits non-zero

---

### REQ-A03: Auth Schema Tables

**Priority:** P0
**Description:** The runtime SHALL define Drizzle schema tables for Better Auth's required data model in `runtime/src/db/schema.ts`: `user`, `session`, `account`, `verification`.

#### Acceptance Criteria
- **GIVEN** `runtime/src/db/schema.ts` exists **WHEN** `drizzle-kit push` runs **THEN** four tables SHALL be created in LibSQL: `user`, `session`, `account`, `verification`. Tables use `auth_` prefix (e.g., `auth_user`, `auth_session`) to avoid conflicts with Mastra's existing tables.
- **GIVEN** the `user` table **WHEN** it is created **THEN** it SHALL contain: `id` (text PK), `name` (text), `email` (text unique), `emailVerified` (integer/boolean), `image` (text nullable), `createdAt` (integer/timestamp), `updatedAt` (integer/timestamp)
- **GIVEN** the `session` table **WHEN** it is created **THEN** it SHALL contain: `id` (text PK), `userId` (text FK→user), `expiresAt` (integer/timestamp), `token` (text unique), `ipAddress` (text nullable), `userAgent` (text nullable), `createdAt`, `updatedAt`
- **GIVEN** the `account` table **WHEN** it is created **THEN** it SHALL contain provider fields (`accountId`, `providerId`, `accessToken`, `refreshToken`, `idToken`, scope, password, etc.)
- **GIVEN** the `verification` table **WHEN** it is created **THEN** it SHALL contain: `id`, `identifier`, `value`, `expiresAt`, `createdAt`, `updatedAt`

#### Scenarios
- **Happy path:** All 4 tables created with correct columns and constraints via `drizzle-kit push`
- **Edge case:** Table already exists with matching schema → drizzle-kit push succeeds (no-op or alter)
- **Error case:** Table exists with incompatible schema → drizzle-kit push fails with schema mismatch error

### REQ-A04: Better Auth Instance with Drizzle Adapter

**Priority:** P0
**Description:** The runtime SHALL export a Better Auth instance from `runtime/src/auth/index.ts` configured with `drizzleAdapter(db, { provider: 'sqlite' })`.

#### Acceptance Criteria
- **GIVEN** `runtime/src/auth/index.ts` is imported **WHEN** it initializes **THEN** it SHALL call `drizzleAdapter(db, { provider: 'sqlite' })` as the database adapter
- **GIVEN** `emailAndPassword.enabled: true` **WHEN** a user submits email+password **THEN** Better Auth SHALL create a user record and session
- **GIVEN** `BETTER_AUTH_SECRET` is at least 32 characters **WHEN** Better Auth starts **THEN** it SHALL use it as the signing secret

#### Scenarios
- **Happy path:** Better Auth starts, generates signed session tokens, creates user/session records in LibSQL
- **Edge case:** `BETTER_AUTH_SECRET` is shorter than 32 chars → Better Auth logs a warning but proceeds
- **Error case:** DB connection fails → Better Auth throws on first request, not on import

### REQ-A05: Session Cookie Configuration

**Priority:** P0
**Description:** Session cookies SHALL be configured for 7-day fixed expiry with HttpOnly, SameSite=Lax, and Secure in production.

#### Acceptance Criteria
- **GIVEN** a user logs in **WHEN** a session is created **THEN** the session cookie SHALL have `expiresIn: 60 * 60 * 24 * 7` (7 days)
- **GIVEN** session cookies **WHEN** they are created **THEN** they SHALL have `httpOnly: true`, `sameSite: 'lax'`, and `secure: true` (only when `NODE_ENV === 'production'`)
- **GIVEN** sliding window is disabled **WHEN** a user makes requests **THEN** the session expiry SHALL NOT be extended (fixed expiry at creation time)

#### Scenarios
- **Happy path:** User logs in, cookie set with 7-day expiry, user can access protected routes for 7 days without re-auth
- **Edge case:** `NODE_ENV=development` → `secure: false` on cookie, session still works on HTTP localhost
- **Error case:** Cookie rejected by browser → check SameSite and Secure attribute compatibility with browser

---

### REQ-A06: trustedOrigins Configuration

**Priority:** P0
**Description:** Better Auth SHALL be configured with `trustedOrigins` including the Caddy origin to allow session cookies to work across the reverse proxy.

#### Acceptance Criteria
- **GIVEN** `BETTER_AUTH_URL` is set to `http://localhost:3001` in development **WHEN** Better Auth initializes **THEN** it SHALL include `http://localhost:3001` in `trustedOrigins`
- **GIVEN** a browser at `http://localhost:3001` makes a request to `/auth/*` **WHEN** the session cookie is set **THEN** it SHALL be visible to the frontend JavaScript only if the origin is in `trustedOrigins`

#### Scenarios
- **Happy path:** Cookie set by `/auth/sign-in` is sent back on subsequent requests from `http://localhost:3001`
- **Edge case:** Production origin is different from localhost → `trustedOrigins` must include production origin
- **Error case:** Origin not in `trustedOrigins` → cookie not sent, session appears logged out

---

### REQ-A07: /auth/* Route Mounting in Mastra

**Priority:** P0
**Description:** The runtime SHALL mount Better Auth's HTTP handler at `/auth/*` on Mastra's server, accessible through the existing Caddy reverse proxy.

#### Acceptance Criteria
- **GIVEN** Mastra starts on port 4111 **WHEN** a POST or GET request arrives at `/auth/*` **THEN** it SHALL be routed to `auth.handler(c.req.raw)`
- **GIVEN** Caddy reverse proxy is configured **WHEN** a browser request goes to `http://localhost:3001/auth/*` **THEN** it SHALL be forwarded to `runtime:4111/auth/*`
- **GIVEN** the browser only talks to the Caddy origin **WHEN** auth requests are made through `/auth/*` **THEN** no dedicated CORS configuration SHALL be required

#### Scenarios
- **Happy path:** POST to `/auth/sign-in` with email+password creates session and returns session cookie
- **Edge case:** Request to `/auth/*` before Better Auth is initialized → returns 500 error from Mastra
- **Error case:** Malformed request body → Better Auth returns appropriate error response

---

### REQ-A08: drizzle-kit push Schema Application

**Priority:** P0
**Description:** The auth schema SHALL be applied to LibSQL using `drizzle-kit push` without generating migration files.

#### Acceptance Criteria
- **GIVEN** `runtime/src/db/schema.ts` contains the auth tables **WHEN** `cd runtime && npx drizzle-kit push` runs **THEN** all four tables SHALL be created in LibSQL
- **GIVEN** the schema was already pushed **WHEN** `drizzle-kit push` runs again **THEN** it SHALL report no changes or apply only deltas

#### Scenarios
- **Happy path:** `drizzle-kit push` completes with zero errors and all tables exist in LibSQL
- **Edge case:** Schema unchanged on subsequent push → succeeds with no changes message
- **Error case:** LibSQL server down → `drizzle-kit push` fails with connection error

---

### REQ-A09: Zod Schemas from Drizzle ORM

**Priority:** P1
**Description:** The runtime SHALL export Zod schemas for auth tables derived from the Drizzle schema using `drizzle-orm/zod`.

#### Acceptance Criteria
- **GIVEN** `runtime/src/db/schema.ts` exists **WHEN** `drizzle-orm/zod` is imported **THEN** it SHALL provide `createSelectSchema` and `createInsertSchema` for each auth table
- **GIVEN** `runtime/src/lib/schemas/auth.ts` exists **WHEN** it is imported **THEN** it SHALL export Zod schemas for `user`, `session`, `account`, `verification`

#### Scenarios
- **Happy path:** Zod schemas validate correctly against valid and invalid input shapes
- **Edge case:** Schema column added/removed → Zod schemas update accordingly
- **Error case:** Invalid Zod schema derivation → TypeScript compilation error

## Technical Approach

### Architecture
```text
Frontend (React :3001)
  └── Caddy reverse proxy
        └── /api/*, /auth/* → runtime (:4111)
              └── Mastra
                    └── /auth/* → Better Auth handler
                          └── LibSQL (sqld :8080)
                                └── user, session, account, verification tables
```

### Packages to Add (runtime/package.json)
```text
better-auth
@better-auth/drizzle-adapter
drizzle-kit
@libsql/client
drizzle-orm
```

### File Map
```text
runtime/src/
  auth/
    index.ts               # Better Auth instance
  db/
    client.ts              # Drizzle singleton
    schema.ts              # Better Auth tables (user, session, account, verification)

runtime/src/lib/schemas/
  auth.ts                  # Zod schemas from Drizzle (createSelectSchema/createInsertSchema)

drizzle.config.ts          # dialect: 'turso'
.env.example               # BETTER_AUTH_SECRET, BETTER_AUTH_URL, LIBSQL_URL
```

### Key Constraints
1. `dialect: 'turso'` in drizzle.config.ts (even for sqld local) — NOT `'sqlite'`
2. `provider: 'sqlite'` in drizzleAdapter — NOT `'turso'`
3. `drizzle-orm/zod` — NOT the deprecated `drizzle-zod` package
4. No authToken in LibSQL connection (self-hosted, no auth)
5. `BETTER_AUTH_URL=http://localhost:3001` (dev browser origin for cookie scope)
6. Same-origin auth only: `/auth/*` proxied by Caddy; no dedicated CORS needed
