# SPEC-20260408-001: Docker Compose Architecture Alignment

## Purpose

Align the existing Docker Compose infrastructure (established by SPEC-20260407-001) with the architecture document's requirements for network segmentation, dev/prod mode switching via docker-compose.override.yml, Caddyfile environment variable switching, frontend runtime config.json serving, and minor health check improvements. This spec covers only the delta — changes needed on top of the already-implemented 9-service stack.

## Scope

### In Scope
- Replacing the single `triage` network with two networks: `app` and `langfuse`
- Runtime service joining both `app` and `langfuse` networks
- New `docker-compose.override.yml` for dev mode (Vite HMR, tsx --watch)
- Caddyfile refactor: env var snippet switching between static (prod) and Vite proxy (dev)
- Caddy serving `/config.json` for frontend runtime configuration
- Adding `CLICKHOUSE_PASSWORD` environment variable to clickhouse service
- MinIO health check alignment with official Langfuse compose (`mc ready local`)
- Redis health check improvement (`--no-auth-warning` flag)
- New environment variables in `.env.example` (`FRONTEND_MODE`, `CADDY_MODE`, `CLICKHOUSE_PASSWORD`)

### Out of Scope
- Adding or removing services (the 9-service set is unchanged)
- Kubernetes Helm chart changes (separate concern)
- Application code, agent logic, or workflow changes
- CI/CD pipeline configuration
- TLS/SSL, DNS, or production deployment
- Changes to Dockerfile.runtime or Dockerfile.frontend build stages (only compose/Caddy config changes)
- The content or schema of config.json itself (only the serving mechanism)

## Requirements

### REQ-A01: Dual Network Segmentation
**Priority:** P0
**Description:** The system SHALL define two Docker networks in docker-compose.yml: `app` (bridge) for frontend, runtime, and libsql services, and `langfuse` (bridge) for langfuse-web, langfuse-worker, clickhouse, redis, minio, and langfuse-postgres services. The existing single `triage` network SHALL be removed.

#### Acceptance Criteria
- **GIVEN** the docker-compose.yml networks section **WHEN** inspected **THEN** it defines exactly two networks: `app` (driver: bridge) and `langfuse` (driver: bridge)
- **GIVEN** services frontend and libsql **WHEN** their network configuration is inspected **THEN** they are attached only to the `app` network
- **GIVEN** services langfuse-web, langfuse-worker, clickhouse, redis, minio, langfuse-postgres **WHEN** their network configuration is inspected **THEN** they are attached only to the `langfuse` network
- **GIVEN** docker compose starts successfully **WHEN** `docker network ls` is run **THEN** both `app` and `langfuse` networks exist

#### Scenarios
- **Happy path:** All services start, each resolves DNS only for services on their own network(s)
- **Edge case:** A service on `app` attempts to connect to a `langfuse`-only service — connection is refused/times out (network isolation works)
- **Error case:** If network definition is missing, compose fails at startup with a clear error

---

### REQ-A02: Runtime Dual-Network Membership
**Priority:** P0
**Description:** The runtime service SHALL be attached to both the `app` and `langfuse` networks so it can receive requests from frontend/Caddy (via `app`) and send traces to langfuse-web (via `langfuse`).

#### Acceptance Criteria
- **GIVEN** the runtime service definition **WHEN** its networks list is inspected **THEN** it includes both `app` and `langfuse`
- **GIVEN** runtime is running **WHEN** it resolves `langfuse-web` by hostname **THEN** DNS resolution succeeds
- **GIVEN** runtime is running **WHEN** it resolves `libsql` by hostname **THEN** DNS resolution succeeds
- **GIVEN** frontend is running **WHEN** Caddy proxies to `runtime:4111` **THEN** the request succeeds (both on `app` network)

#### Scenarios
- **Happy path:** Runtime sends OTEL traces to langfuse-web:3000 and serves API requests from Caddy — both work simultaneously
- **Edge case:** Langfuse services are down — runtime still starts and serves API requests on the `app` network
- **Error case:** If runtime is only on `app`, trace export to langfuse-web fails with connection refused

---

### REQ-A03: Docker Compose Override for Dev Mode
**Priority:** P0
**Description:** The system SHALL include a `docker-compose.override.yml` that auto-loads during development (`docker compose up`). The override SHALL: (a) add a `vite` service running the Vite dev server with HMR on port 5173, (b) replace the runtime CMD with `tsx --watch` for hot-reload, (c) mount source code volumes for both frontend and runtime, and (d) set `FRONTEND_MODE=dev` for Caddy. Production mode SHALL be invoked with `docker compose -f docker-compose.yml up --build` to skip the override.

#### Acceptance Criteria
- **GIVEN** a developer runs `docker compose up` **WHEN** docker-compose.override.yml exists **THEN** it is auto-loaded and dev services start with HMR and watch mode
- **GIVEN** a developer runs `docker compose -f docker-compose.yml up --build` **WHEN** only the base file is specified **THEN** the override is NOT loaded and production mode starts
- **GIVEN** the override is loaded **WHEN** the vite service starts **THEN** it exposes port 5173 and serves the frontend with HMR
- **GIVEN** the override is loaded **WHEN** the runtime service starts **THEN** it runs with tsx --watch instead of the production CMD
- **GIVEN** the override is loaded **WHEN** source files change **THEN** both vite and runtime detect changes and hot-reload

#### Scenarios
- **Happy path:** Developer runs `docker compose up`, edits a .tsx file, sees changes reflected in browser without manual restart
- **Edge case:** Developer has no override file (deleted or CI environment) — base compose works standalone in production mode
- **Error case:** Volume mount paths are incorrect — container starts but source files are missing; error is visible in logs

### REQ-A04: Caddyfile Environment Variable Switching
**Priority:** P0
**Description:** The Caddyfile SHALL use Caddy's `{$ENV_VAR:default}` syntax and snippet imports to switch between static file serving (production) and Vite reverse proxy (development). The system SHALL define two snippets: `(static-frontend)` for production (root /srv, try_files, file_server) and `(dev-frontend)` for development (reverse_proxy vite:5173). The active snippet SHALL be selected via `import {$FRONTEND_MODE:static}-frontend`. The default MUST be `static` (production) when `FRONTEND_MODE` is not set.

#### Acceptance Criteria
- **GIVEN** FRONTEND_MODE is unset or set to `static` **WHEN** Caddy starts **THEN** it serves static files from /srv with SPA fallback
- **GIVEN** FRONTEND_MODE is set to `dev` **WHEN** Caddy starts **THEN** it reverse-proxies to vite:5173
- **GIVEN** the Caddyfile **WHEN** inspected **THEN** it contains two named snippets `(static-frontend)` and `(dev-frontend)` and uses `import {$FRONTEND_MODE:static}-frontend`
- **GIVEN** production mode **WHEN** no FRONTEND_MODE env var exists **THEN** Caddy defaults to static file serving (safe default)

#### Scenarios
- **Happy path (prod):** FRONTEND_MODE unset, Caddy serves built SPA from /srv, API proxied to runtime:4111
- **Happy path (dev):** FRONTEND_MODE=dev, Caddy proxies frontend to vite:5173, API still proxied to runtime:4111
- **Edge case:** FRONTEND_MODE set to invalid value — Caddy fails to start with clear error (snippet not found)
- **Error case:** Vite service not running but FRONTEND_MODE=dev — Caddy starts but returns 502 for frontend routes

### REQ-A05: Frontend Runtime Config.json
**Priority:** P0
**Description:** Caddy SHALL serve a `/config.json` endpoint that provides frontend runtime configuration (API URL, feature flags). This file SHALL be served before SPA fallback so it is not caught by the `try_files {path} /index.html` rule. In production, `config.json` SHALL be a static file mounted or placed at `/srv/config.json`. In dev mode, the Vite dev server handles this route.

#### Acceptance Criteria
- **GIVEN** production mode **WHEN** a client requests `GET /config.json` **THEN** Caddy returns the config.json file with `Content-Type: application/json`
- **GIVEN** production mode **WHEN** config.json is updated on disk **THEN** the next request returns the updated content without rebuilding the frontend image
- **GIVEN** dev mode **WHEN** a client requests `GET /config.json` **THEN** the request is proxied to Vite which serves it from the public directory
- **GIVEN** docker-compose.yml **WHEN** the frontend service is inspected **THEN** config.json is volume-mounted (not baked into the image)

#### Scenarios
- **Happy path:** Frontend SPA fetches /config.json on startup, receives JSON with API base URL and feature flags
- **Edge case:** config.json does not exist — Caddy returns 404 (not the SPA index.html)
- **Error case:** config.json contains invalid JSON — Caddy still serves it (not its job to validate), frontend handles parse error

### REQ-A06: ClickHouse Password Configuration
**Priority:** P1
**Description:** The clickhouse service SHALL have `CLICKHOUSE_PASSWORD` set via environment variable, aligning with the Langfuse official compose configuration. The password SHALL be defined in `.env.example` with a CHANGEME placeholder.

#### Acceptance Criteria
- **GIVEN** the clickhouse service definition **WHEN** its environment section is inspected **THEN** `CLICKHOUSE_PASSWORD` is set from the env var
- **GIVEN** .env.example **WHEN** inspected **THEN** it contains `CLICKHOUSE_PASSWORD=CHANGEME`
- **GIVEN** langfuse-web and langfuse-worker services **WHEN** their CLICKHOUSE_PASSWORD env var is inspected **THEN** it matches the clickhouse service password

#### Scenarios
- **Happy path:** ClickHouse starts with password set, Langfuse connects successfully using the same password
- **Edge case:** Password is empty string — ClickHouse still starts (permissive default)
- **Error case:** Password mismatch between ClickHouse and Langfuse services — Langfuse logs connection auth errors

### REQ-A07: MinIO Health Check Alignment
**Priority:** P2
**Description:** The MinIO health check SHOULD use `mc ready local` to align with the official Langfuse compose. This requires the `mc` (MinIO Client) binary to be available in the MinIO container image.

#### Acceptance Criteria
- **GIVEN** the minio service definition **WHEN** its healthcheck is inspected **THEN** it uses `mc ready local` (or equivalent `curl` fallback if mc is unavailable)
- **GIVEN** minio is healthy **WHEN** dependent services start **THEN** they can connect to MinIO successfully

#### Scenarios
- **Happy path:** MinIO starts, `mc ready local` returns success, dependents start
- **Edge case:** MinIO image does not include `mc` binary — fall back to `curl -f http://localhost:9000/minio/health/ready`
- **Error case:** MinIO fails to start — health check fails, dependents wait

### REQ-A08: Redis Health Check Auth Warning Suppression
**Priority:** P2
**Description:** The Redis health check SHOULD use the `--no-auth-warning` flag to suppress the password-in-process-list warning: `redis-cli --no-auth-warning -a ${REDIS_AUTH} ping`.

#### Acceptance Criteria
- **GIVEN** the redis service healthcheck **WHEN** inspected **THEN** it includes `--no-auth-warning` flag
- **GIVEN** redis is running **WHEN** the health check executes **THEN** no auth warning appears in logs

#### Scenarios
- **Happy path:** Health check runs silently, returns PONG, redis marked healthy
- **Edge case:** REDIS_AUTH is empty — redis-cli ping works without -a flag (but current config always sets auth)
- **Error case:** Redis is down — health check returns non-zero, container marked unhealthy

### REQ-A09: Environment Variable Updates
**Priority:** P0
**Description:** The `.env.example` file SHALL be updated to include new environment variables: `FRONTEND_MODE=static` (with comment explaining dev/static options), `CLICKHOUSE_PASSWORD=CHANGEME`. Existing variables SHALL NOT be removed or reordered.

#### Acceptance Criteria
- **GIVEN** .env.example **WHEN** inspected **THEN** it contains `FRONTEND_MODE=static` with a comment explaining valid values
- **GIVEN** .env.example **WHEN** inspected **THEN** it contains `CLICKHOUSE_PASSWORD=CHANGEME`
- **GIVEN** .env.example **WHEN** compared to the previous version **THEN** no existing variables are removed or renamed

#### Scenarios
- **Happy path:** Developer copies .env.example to .env, all services start with defaults
- **Edge case:** Developer sets FRONTEND_MODE=dev without override file — Caddy tries to proxy to vite:5173 which doesn't exist (502 errors, but system doesn't crash)
- **Error case:** CLICKHOUSE_PASSWORD not set — ClickHouse uses empty password (insecure but functional)

## Technical Approach

This spec makes targeted modifications to existing infrastructure files established by SPEC-20260407-001. The key architectural decisions are:

1. **Network segmentation** follows defense-in-depth: the `app` network isolates user-facing services from observability infrastructure on the `langfuse` network. The runtime service bridges both networks since it must serve API requests and export traces. This is a standard Docker Compose multi-network pattern.

2. **Dev/prod mode switching** uses Docker Compose's built-in override mechanism: `docker-compose.override.yml` auto-loads when present, and production explicitly specifies only the base file. This avoids conditional logic in the base compose file.

3. **Caddyfile snippet switching** leverages Caddy's native `{$ENV_VAR:default}` substitution to dynamically import the correct snippet at parse time. This is more robust than conditional blocks (which Caddy doesn't support) and avoids maintaining two separate Caddyfiles.

4. **config.json** is served as a static file by Caddy, volume-mounted rather than baked into the image. This enables environment-specific configuration without rebuilding the frontend. The handle directive order in Caddy ensures `/config.json` is served before the SPA catch-all.

5. **ClickHouse password** and **health check improvements** are minor alignment items that bring the implementation closer to Langfuse's official reference compose.

## Test Strategy

### Infrastructure Tests (Shell/Docker)
- **Network isolation test:** Start stack, exec into frontend container, verify it cannot resolve `clickhouse` hostname. Exec into runtime, verify it can resolve both `libsql` and `langfuse-web`.
- **Dev mode test:** Start with `docker compose up`, verify vite service is running and Caddy proxies to it. Start with `docker compose -f docker-compose.yml up`, verify no vite service and Caddy serves static files.
- **Config.json test:** `curl http://localhost:3001/config.json` returns valid JSON in both dev and prod modes.
- **Health check tests:** Verify all services reach healthy state within 60 seconds.
- **ClickHouse auth test:** Verify langfuse-web can query ClickHouse with configured password.

### Validation Approach
- `docker compose config` validates the merged compose files parse correctly
- `caddy validate --config Caddyfile` validates Caddyfile syntax
- `docker compose --profile dev config` validates override merging
- Manual smoke test: full stack up, frontend loads, API responds, traces appear in Langfuse

### No Unit Tests
This spec modifies only infrastructure configuration (YAML, Caddyfile, env files). There is no application code to unit test. Validation is via compose parsing, container health checks, and integration smoke tests.

## Implementation Plan

1. **Update docker-compose.yml networks:** Replace the single `triage` network with `app` and `langfuse` networks. Assign each service to the appropriate network. Assign runtime to both networks.

2. **Update Caddyfile:** Refactor into snippets (`static-frontend`, `dev-frontend`). Add `import {$FRONTEND_MODE:static}-frontend` directive. Ensure `/config.json` is served correctly (handle order before SPA catch-all). Preserve existing security headers and API proxy config.

3. **Create docker-compose.override.yml:** Define vite service (node:22-alpine, Vite dev server, port 5173, source volume mount). Override runtime CMD to tsx --watch. Set FRONTEND_MODE=dev for frontend/Caddy. Mount source volumes for hot-reload.

4. **Update clickhouse service:** Add `CLICKHOUSE_PASSWORD` environment variable. Ensure langfuse services reference the same password variable.

5. **Update MinIO health check:** Change from curl to `mc ready local` (with comment noting curl fallback).

6. **Update Redis health check:** Add `--no-auth-warning` flag.

7. **Update .env.example:** Add `FRONTEND_MODE=static`, `CLICKHOUSE_PASSWORD=CHANGEME` with comments.

8. **Create config.json template:** Add a sample `config.json` file and volume mount in docker-compose.yml for the frontend service.

9. **Validate:** Run `docker compose config` on both base and merged (with override) configurations. Run `caddy validate`. Start full stack and verify all health checks pass.
